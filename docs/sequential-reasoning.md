# Feature: Sequential Reasoning Engine (Sub-Agent Orchestration)
**Status:** Approved for Implementation  
**Target Release:** v1.4.0  
**Epic:** Enhanced Inference Capabilities & Enterprise Reasoning  

## 1. Goal
Transform the Beexexity gateway from a single-turn LLM proxy into a transparent, sequential reasoning engine. For highly complex, multi-faceted requests, the system will dynamically generate a step-by-step Chain-of-Thought execution plan. Instead of disjointed parallel agents, the system executes deliberate, sequential steps where each step builds upon the context of the previous one, culminating in a highly cohesive, narrative-driven final response. 

This feature strictly adheres to **Transparency**, **Narrative Cohesion**, and **Cost-Efficiency** principles: it only triggers when necessary, provides real-time visibility into the AI's "thinking process" via SSE, and avoids the massive token multiplication of parallel execution.

## 2. User Stories
- **As a** banking analyst, **I want** the gateway to methodically break down my complex portfolio risk query step-by-step, **so that** I receive a cohesive, logically flowing report rather than a disjointed collection of parallel outputs.
- **As a** compliance officer, **I want** to see real-time UI updates showing the AI's sequential progress (e.g., *"Cross-referencing against BI regulations..."*), **so that** I trust the depth of the analysis and know the system is not stuck.
- **As a** user uploading a massive 50-page PDF, **I want** the system to automatically distill the document in the first step before analyzing it, **so that** the subsequent analytical steps remain highly focused and accurate.
- **As a** user, **I want** my follow-up questions to perfectly remember the step-by-step analysis from the previous turn, **so that** the conversation flows naturally without losing context.

## 3. Acceptance Criteria (BDD)

### Scenario 1: Strict Triggering & Guardrails
- **GIVEN** a user input is received
- **WHEN** the `RoutingEngine` evaluates the request
- **THEN** the reasoning engine triggers **ONLY IF**:
  1. `skill` is in `[compliance_pre_assessment, requirement_generation, document_qna]`
  2. `complexity_score >= 4`
  3. `originalPrompt.length >= 120` (unless `hasImages` is true)
- **AND** the Planner LLM returns a valid sequential plan with `steps.length > 1` and `steps.length <= 10`. If the Planner returns 1 step, the system **MUST** fall back to standard single-shot `generate()`.

### Scenario 2: Sequential Execution & Radical Transparency
- **GIVEN** a valid sequential plan is generated by the Planner
- **WHEN** the Executor begins the loop
- **THEN** it MUST execute the steps strictly sequentially (Step 1 → Step 2 → Step N).
- **AND** before executing *each* step, it MUST emit an `orchestration_status` SSE event with a human-readable description (e.g., *"Step 2 of 4: Cross-referencing extracted data against Bank Indonesia regulations..."*).
- **AND** each step receives the output of the previous step as accumulated context, ensuring logical continuity.
- **AND** the final step acts as the Synthesizer, formatting the accumulated context into a cohesive, conversational final response.

### Scenario 3: Large File "Map-Reduce" Preprocessing
- **GIVEN** a user uploads a document where `maskedDocumentText.length > 30,000` characters
- **WHEN** the Planner generates the execution plan
- **THEN** Step 1 MUST automatically be assigned as the "Data Cruncher" (using `qwen3-235b`).
- **AND** the Data Cruncher's sole output is a condensed, structured summary (~2,000 chars).
- **AND** Steps 2 through N will inject this *condensed summary* instead of the raw 30k+ chars, preventing context window exhaustion.

### Scenario 4: Quality & Chain-of-Thought (CoT)
- **GIVEN** a sequential step is executing
- **THEN** the system prompt for that step MUST enforce internal Chain-of-Thought reasoning (e.g., using `<thinking>` tags) before generating the step's final output.
- **AND** if a step identifies a contradiction or risk, it passes this explicitly to the next step. The final step MUST resolve these contradictions into a definitive recommendation, rather than just highlighting them.

### Scenario 5: Multi-Turn Continuity (Context Amnesia Fix)
- **GIVEN** a sequential reasoning turn (Turn N) is completed
- **WHEN** the user sends a follow-up (Turn N+1)
- **THEN** the context assembler **MUST** inject a hidden `<orchestration_context>` metadata block into the prompt for Turn N+1 **only**.
- **AND** the Tier 2 `rolling_summary` and Tier 3 `extracted_facts` generators **MUST** use specialized prompts to extract the specific risks, findings, and final verdicts from the sequential steps.

### Scenario 6: PII Safety & Fire-and-Forget Auditing
- **GIVEN** sequential steps are executing
- **THEN** the `pii-masker.service.ts` **MUST** be applied to the input and output of *every single step*.
- **AND** the `audit.service.ts` **MUST** execute a fire-and-forget `logInference()` call for every step (Planner + Step 1 to N), ensuring 100% of Bedrock costs are visible in the billing system.

## 4. Non-Goals
- **Parallel Execution:** Sub-agents will not run concurrently (`Promise.allSettled` is deprecated for this feature). Sequential execution ensures narrative cohesion.
- **External Tool Calling:** No function/tool definitions will be passed to Bedrock. Steps are purely prompt-based reasoning.
- **Human-in-the-loop:** No pausing for user feedback mid-orchestration. The plan executes to completion.

## 5. Success Metrics

- **Accuracy & Cohesion**: Drastic reduction in "Frankenstein" reports. Final outputs must read as a single, cohesive narrative rather than stitched-together segments. Hallucinations reduced via step-by-step Chain-of-Thought.
- **Latency**: 
  - Standard sequential turns (3-4 steps): P95 latency remains under **30 seconds**.
  - Deep-dive turns (up to 10 steps) or Large Files: P95 latency accepted up to **60 seconds**, fully mitigated by real-time SSE transparency.
- **Cost**: Overall token cost increase is capped at **4x** the cost of a standard single-shot inference (reduced from the previous 6x parallel cap, as sequential execution avoids token multiplication).
- **Continuity**: 100% of follow-up questions referencing a previous deep-dive successfully retrieve the specific findings without requiring the user to re-upload the document.
- **Transparency & Observability**:
  - **User-Facing Progress:** 100% of orchestrated turns emit granular `orchestration_status` SSE events for *every single step*. Users see a live, step-by-step progress feed (e.g., *"Step 1/4: Condensing document..."*, *"Step 2/4: Extracting financial clauses..."*), completely eliminating the "stuck" perception.
  - **Developer-Facing Debugging:** 100% of sequential steps are logged via fire-and-forget auditing. This enables precise debugging, prompt iteration (tuning the Planner and individual step prompts), and exact cost attribution.
